# Модуль 8: Мониторинг и наблюдаемость ML-сервиса в продакшене

Проект направлен на создание полноценной системы наблюдаемости (Observability) для ML-сервиса онлайн-кинотеатра, включающего подсистему рекомендаций и генеративный сервис Virtual Product Placement (VPP — встраивание рекламы в видео).

---

## 🎯 Определенные SLO (Service Level Objectives)

На основе анализа противоречивых требований бизнеса, разработки, ML и DevOps (через сбалансированное дерево метрик), были приняты следующие SLO в формате OpenSLO:

1. **Латентность (p95):** `< 200ms` для 99.9% запросов инференса.
    - _Обоснование:_ Бизнес требует "чтобы быстро было", DevOps указывает на накладные расходы Docker-in-Docker.
2. **Доступность:** `99.9%` (Отказ от нереалистичного требования бизнеса в 100% аптайма).
3. **Пропускная способность:** `10 000 RPS` при условии использования кэша (решение проблемы узкого места Postgres, который выдает только 1000 RPS без кэша).

---

## 🛠 Шаги настройки и реализации

### Шаг 1: Стратегия метрик и ADR

- **Действие:** Сформировано 4-уровневое дерево метрик (Business -> SLI/SLO -> DevOps -> ML). Создан файл `ADR-0001` (Architecture Decision Record) с фиксацией рисков (например, нехватка GPU, узкое место DinD Postgres).
- **Инструменты:** `graphviz` (для диаграммы), Markdown (для симуляции `adr-tools`).

### Шаг 2: Инфраструктурный мониторинг (Prometheus & Grafana)

- **Действие:** Выбрана ключевая метрика — `p95 Inference Latency`. Настроен сбор гистограмм (`_bucket` метрик). В Grafana реализован провижининг дашборда через YAML и JSON, настроен расчет `histogram_quantile(0.95, ...)` и визуализация "Error Budget" (Бюджета ошибок).
- **Артефакты:** `prometheus.yaml`, `grafana.yaml`, `ml-slo-dashboard.json`.

### Шаг 3: Обнаружение деградации и дрифта (Evidently)

- **Действие:** Смоделирован **Data Drift**. В качестве Reference batch использованы данные привычной аудитории (средний возраст 35 лет, сессия 2 часа). В Current batch поданы данные новой аудитории (возраст 20 лет, короткие сессии).
- **Результат:** Evidently Report наглядно показал смещение распределений (красные зоны) по признакам `user_age` и `session_duration_min`.

### Шаг 4: Data Quality Ops (DQOps)

- **Действие:** Развернут контур DQOps + PostgreSQL + туннелирование портов (Tuna). В БД создана таблица `user_recommendations`.

### Шаг 5: Архитектура ML-системы VPP (Kappa Architecture)

- **Действие:** Спроектирована система встраивания логотипов на футболки персонажей в видеопотоке. Выбрана **Kappa-архитектура** (вместо Lambda), так как вся обработка идет в реальном времени через стримы.
- **Стек:** `FastStream` + `Redis Streams` (брокер сообщений) + `YOLOv11` (детекция человека и области для логотипа).
- **Инструменты:** Библиотека `diagrams` (с исправленным импортом `Redis` из `diagrams.onprem.inmemory`).
- **Артефакты:** Код генерации архитектурной диаграммы в формате `.png`.
